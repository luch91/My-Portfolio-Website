A PM's Deep Dive into RAG Systems

RAG is usually described as "retrieve documents, send to LLM, get answer." That description hides exactly where systems fail in production. Here is what the pipeline actually looks like and what it means for product decisions.

Why the Simple Description Is Not Enough

RAG is usually described as: retrieve documents, send to LLM, generate answer.

That description hides where systems fail in production. A real RAG system is a multi-stage pipeline with independent failure modes, latency constraints, and evaluation challenges. Understanding those layers is what separates feature PMs from platform PMs.

Full System Overview

A production-grade RAG system is better understood as seven distinct stages, each with its own failure surface.

RAG Production Pipeline — 7 Stages

Each stage introduces trade-offs. The rest of this article breaks each one down.

LAYER 01 Query Processing

Purpose: Transform raw user input into a structured query optimised for retrieval.

Components

  • Query rewriting (LLM-assisted or rule-based)
  • Intent classification
  • Entity extraction
Query Rewriting
Failure Modes
  • Loss of intent nuance during rewriting
  • Over-generalisation that broadens the query too much
  • Incorrect entity extraction
PM Insight

Query processing directly impacts retrieval quality. Weak rewriting cascades downstream: every subsequent stage inherits the error.

LAYER 02 Retrieval (the Core)

Purpose: Fetch the most relevant information from the knowledge base.

Architecture Options

  • Dense retrieval: embeddings via sentence transformers, then vector similarity search (FAISS, Pinecone, Qdrant)
  • Sparse retrieval: keyword-based BM25
  • Hybrid retrieval: combine both, then re-rank
Hybrid Retrieval Architecture

Key Decisions

  • Embedding model choice
  • Chunk size: too small loses context; too large reduces precision
  • Top-K retrieval size
  • Re-ranking strategy
Failure Modes
  • Irrelevant but semantically similar chunks returned
  • Missing critical context because the chunk boundary was wrong
  • Outdated or low-quality data in the knowledge base
PM Insight

Retrieval quality sets the upper bound of system accuracy. The LLM cannot recover missing knowledge. This is the layer to invest in first.

LAYER 03 Context Construction

Purpose: Assemble retrieved chunks into a structured prompt that the LLM can reason over.

Considerations

  • Token limits: you cannot fit everything
  • Ordering of context chunks (recency vs. relevance)
  • Deduplication of overlapping chunks
  • Metadata inclusion: source name, date, confidence score
Prompt Assembly Under Token Constraints
Failure Modes
  • Context overflow causing truncation of high-value chunks
  • Wrong ordering reducing LLM attention on key passages
  • Conflicting information with no resolution signal
PM Insight

Context construction is an information compression problem under hard constraints. The quality of this step determines what the model can and cannot reason about.

LAYER 04 LLM Inference

Purpose: Generate a response grounded in the retrieved context.

Key Decisions

  • Model selection: latency vs. reasoning quality
  • Prompt design and instruction clarity
  • Temperature and decoding strategy
System: You are a clinical assistant. Use ONLY the provided context. Cite sources. If unsure, say so explicitly. Context: [assembled chunks from Layer 03] Question: {user_query}
Failure Modes
  • Hallucination despite relevant context being present
  • Ignoring provided documents and falling back on parametric memory
  • Shallow reasoning on complex multi-step queries
PM Insight

Model choice affects cost per request, latency, and reasoning depth simultaneously. This is a product and infrastructure trade-off, not just a model decision.

LAYER 05 Post-processing & Guardrails

Purpose: Ensure every response meets safety and quality standards before delivery.

Components

  • Output validation (schema, format, length)
  • Toxicity filtering
  • Rule-based constraints (domain-specific blocklists)
  • Citation verification
Failure Modes
  • Unsafe recommendations passing through undetected
  • Incorrect or fabricated citations
  • Over-filtering that removes genuinely helpful responses
PM Insight

Guardrails are not optional in high-stakes domains (healthcare, finance, legal). The cost of a failure is asymmetric. Design for the worst-case output, not the average one.

LAYER 06 Evaluation (the Most Underrated Stage)

Purpose: Measure system performance and pinpoint exactly where failures occur.

Evaluation Types

  • Retrieval: Recall@K, Precision@K, MRR
  • Generation: answer correctness, citation accuracy, hallucination rate
  • End-to-end: task success rate, user satisfaction
Evaluation Pipeline

Failure Categories to Track

  • Retrieval miss: the right chunk was never fetched
  • Reasoning error: chunks were present but the LLM misinterpreted them
  • Hallucination: model invented information not in context
  • Incomplete answer: query only partially addressed
PM Insight

Evaluation must isolate where the system failed, not just that it failed. A single quality score is too coarse to drive any meaningful improvement decision.

LAYER 07 Feedback Loop & Iteration

Purpose: Continuously improve system performance after deployment using live signals.

Signal Sources

  • User feedback: thumbs, corrections, escalations
  • System logs: latency, retrieval scores, refusals
  • Evaluation datasets: curated ground truth

Actions

  • Improve chunking strategy
  • Refine prompts based on failure analysis
  • Update and refresh embeddings
  • Retrain or swap re-rankers
PM Insight

RAG systems improve through data and iteration loops, not one-time tuning. The product roadmap for an AI system is a sequence of data flywheels. Build the loop on day one.

System Trade-offs: Where PMs Add Value

Strong PMs make these trade-offs explicit rather than leaving them to engineering intuition.

DimensionTrade-off
Latency Deeper retrieval (better answers) vs. faster response (better UX)
Cost Larger models (stronger reasoning) vs. cheaper inference (lower unit cost)
Accuracy More context in the prompt vs. token limits and cost constraints
Freshness Real-time data ingestion vs. cached embeddings (cheaper, but stale)

What Most Teams Get Wrong

  1. Treating RAG as a feature, not a system. It has independent failure modes at every stage. One "RAG ticket" will not cover it.
  2. Ignoring retrieval quality and blaming the model. Most RAG failures trace back to what was (or was not) retrieved, not how the LLM reasoned.
  3. Lacking structured evaluation. You cannot improve what you cannot measure. User thumbs-down is not an evaluation pipeline.
  4. Optimising prompts instead of fixing data. Prompt engineering is the last lever, not the first. Fix the knowledge base, fix chunking, then refine the prompt.

Closing

RAG is not really about making LLMs smarter. It is about building a system where:

  • Retrieval provides the right knowledge
  • The model reasons over it correctly
  • Evaluation ensures reliability
  • Iteration improves performance over time

That system thinking is what defines an AI Platform Product Manager.
