I Built a Full-Stack AI Evaluation Platform for a Clinical Pediatric Nutrition Assistant

AI builders celebrate the demo. The retrieval looks good. The response sounds right. The stakeholders nod. Then a user asks an edge case question and the system confidently returns wrong clinical information. Nobody catches it because nobody built a system to catch it. I built that system.

What CPNA Does

CPNA is a chat-first clinical nutrition assistant for pediatric use cases. It handles four distinct query types.

CPNA Query Type System

The system runs on an agentic RAG architecture: intent classification via DistilBERT, slot extraction, a therapy gatekeeper, hybrid retrieval combining vector search and BM25, passage reranking, deterministic nutrient computation, and response synthesis. 11 supported therapy conditions: Type 1 diabetes, cystic fibrosis, food allergy, preterm nutrition, chronic kidney disease, PKU, MSUD, galactosemia, epilepsy and ketogenic therapy, IBD, and GERD. The pipeline connects to the live CPNA API.

Why This Needed a Dedicated Eval Platform

CPNA outputs therapy-grade nutrition targets for sick children. A hallucinated nutrient target, a missed drug-nutrient interaction, or a wrong downgrade reason are patient safety problems. You cannot eyeball clinical AI safety.

Agentic systems fail at different layers in different ways. A response can pass a surface-level quality check while leaking internal schema objects into the UI. A retrieval pipeline can appear to work while returning evidence from the wrong condition. A context inheritance check can pass while stale patient data bleeds into a new query. You need instrumented, automated checks at every layer running simultaneously.

Wearing PM, designer, architect, and evaluator hats on the same product, I needed to move fast without breaking clinical correctness. Without automated evaluation, every code change required manual end-to-end testing of every scenario.

LAYER The Architecture

The platform has five layers.

CPNA Eval Platform: Five-Layer Architecture

The Five Evaluator Modules

  • Intent evaluator: route accuracy, confidence threshold compliance, fallback on low-confidence inputs, misroute prevention.
  • Gatekeeper evaluator: verifies all 8 required slots before therapy output. Verifies correct downgrade. Verifies no nutrient targets without a gatekeeper pass. Checks unsupported diagnoses trigger graceful downgrade, not hard errors.
  • Contract validator: every API response must match its typed display contract. Scans for raw object leaks, machine field names, placeholder strings like "Food A / Food B," empty sections, missing required fields.
  • Context safety evaluator: checks for silent context contamination. Verifies loose reply resolution. Verifies state resets on new tasks.
  • Retrieval evaluator: hybrid pipeline ran, top-k within limit of 7, evidence matches only what was retrieved that turn, no duplicate reference IDs.

The 10 E2E Scenarios

These are not happy-path tests. Each one probes a specific behavior that has a specific way of failing in a clinical agentic system.

E2E Scenario Coverage Map
E2E-2
Therapy Gated: Missing Medications and Biomarkers
"Therapy plan for 8-year-old girl with cystic fibrosis. Age, sex, weight, height, diagnosis, country provided. Medications and lab values absent."
Why it matters

Cystic fibrosis nutrition management involves enzyme replacement therapy, fat-soluble vitamin status, and pulmonary function. Generating a plan without knowing the child's medications and lab values could produce targets that conflict with active treatment.

Pass Criteria
  • Downgrade triggered with reason "missing_required_slots"
  • Missing slots identified as medications and biomarkers only
  • User-facing explanation names exactly what is needed
  • No nutrient targets in response
E2E-3
Therapy Full Pass: PKU, All 8 Slots
"7-year-old girl, PKU, weight 22kg, height 118cm, phenylalanine 8mg/dL, no medications, Nigeria."
Why it matters

PKU requires precision nutrition from infancy. The phenylalanine tolerance window is narrow and age-weight dependent. A wrong calculation can cause neurological damage. This scenario verifies that the deterministic nutrient engine, not the LLM, is generating the targets.

Pass Criteria
  • Therapy route confirmed, all 8 slots validated
  • nutrient_targets present with phenylalanine restriction and tyrosine supplementation
  • patient_summary present, evidence_summary non-empty
  • No raw schema objects in response
E2E-7
Loose Reply Resolution
Assistant asks for the child's age. User replies: "10."
Why it matters

Real clinical conversations do not come in perfectly structured JSON. If loose replies fail to resolve, the system gets stuck in infinite slot loops or misroutes the conversation entirely.

Pass Criteria
  • Resolved slot = age, resolved value = 10
  • Workflow continues correctly
  • last_user_reply_resolution in state_update reflects correct resolution

How I Built It: The Prompt Pack Approach

I built this with Claude Code using a structured prompt pack: 16 prompts across 6 phases, each prompt a self-contained Claude Code session with a clear deliverable and a verification command.

16-Prompt Build Structure: 6 Phases

The foundation is a CLAUDE.md file committed to the repo root. This is the most important document in the project. It encodes the domain rules every Claude Code session must respect: therapy outputs must never be generated without a confirmed gatekeeper pass; raw backend objects must never reach the UI; context inheritance must be explicit and traceable; evidence used must match only what was retrieved that turn; comparison output must show actual entity names.

Key Insight

Every Claude Code session reads CLAUDE.md before touching code. This is what keeps a 16-prompt build coherent across sessions. Each prompt is scoped, verified, and committed before the next begins.

What the Live Platform Revealed

Frame-by-frame analysis of 60 frames at 5-second intervals across the 5-minute recording found six things automated checks missed entirely.

FINDING 01
A clinical accuracy failure invisible to automated checks

The Gatekeeper Downgrade Event Log showed E2E-7 triggering a downgrade with reason "unsupported_condition" and a user-facing explanation telling the user their condition was not supported. The diagnosis in E2E-7 is epilepsy/ketogenic. Epilepsy/ketogenic is a supported condition listed in the platform's own settings page.

The automated test passed. The check was: did the downgrade fire? Yes. Pass. The check was not: is the downgrade reason correct? A clinician reading that explanation would believe the platform cannot help with epilepsy nutrition management. That is false.

FINDING 02
A scenario not testing its own specification

E2E-2 is designed to test "Therapy Gated: Missing Medications and Biomarkers." The slot heatmap showed all 7 non-diagnosis slots as red, not just the 2 that should be absent. The scenario input was a vague natural language message that did not explicitly provide age, sex, weight, height, or country. The test was structured to fail at the wrong layer. Root cause: scenario definition, not the CPNA system.

FINDING 03
Fixed confidence values

71 classifications with every general, recommendation, and comparison intent at exactly 85% confidence. Therapy at 87%. Zero variance across 71 diverse clinical queries from a live DistilBERT classifier is statistically impossible. The confidence values were not being passed through from the CPNA API. The threshold check was passing on every run because it was checking a constant against a threshold.

FINDING 04
Fixed retrieval relevance

Every scenario showed exactly 0.850 mean relevance. Same root cause: relevance scores were not flowing through from the retrieval layer. The retrieval quality signal had zero information content.

FINDING 05
An inverted date filter

The Logs page FROM date filter was showing results before the selected date rather than after it. One character in the Supabase query: .lte instead of .gte. Invisible in development, undetectable by unit tests, immediately visible when you look at the output.

FINDING 06
Two pages with blank loading states

The Logs page and Context Safety page showed completely blank content areas during data fetch. Section headers rendered. Table headers rendered. The data area was blank white space. No skeleton loaders.

These findings came from watching 300 seconds of screen recording at 5-second resolution and reading what the platform actually produced.

The Eval Surfaces Worth Building

Gatekeeper Slot Heatmap (Concept)
  • The Downgrade Event Log. Every downgrade with missing slots, reason code, and user-facing explanation. The reason code tells you if the logic is correct. The user-facing explanation tells you if a clinician would misunderstand. These are different things.
  • The Retrieval Stage Inspector. Vector count, BM25 count, post-merge count, top-k, and mean relevance per scenario. Comparison scenarios pull more evidence (14 vector, 14 BM25) than single-condition queries (7, 7). The deduplication step is visible.
  • The Context Safety Inheritance Flow. Prior context type, current query type, safe or contaminated. Silent context contamination is one of the most dangerous failure modes in a multi-turn agentic system.
  • The Logs Explorer. 228 entries across four types: intent, context, retrieval, downgrade. Filterable by run, date, and query type. After a failed scenario, you filter to that run and understand exactly what the system did and why.

What This Taught Me

Build eval as a layer, not a phase

Evaluation is infrastructure, not a testing stage before launch. Every new CPNA feature generates new eval scenarios and new checks. The eval platform grows with the product because they are built in parallel.

Automated checks confirm that a behavior occurred. Human review confirms that the behavior was correct.

The epilepsy downgrade case makes this concrete. The check asked: did the downgrade fire? It did. Pass. The human review asked: is the downgrade reason accurate? Is the user-facing explanation safe for a clinician to read? Those questions cannot be answered by a boolean check.

Prompt packs produce structured output. Ad-hoc prompting produces ad-hoc output.

16 prompts across 6 phases, each with a verification command, each reading CLAUDE.md first. The CLAUDE.md file is what keeps each session oriented. It encodes the domain rules, naming conventions, architectural constraints, and the things Claude Code must never do. Every session inherits that context automatically.

Sit down and watch your product.

Build the automation. Run the automation. Then watch 5 minutes of your product at 5-second resolution and read everything it displays. The automation tells you which checks passed. The watching tells you what your users actually see.

What Is Next

  • E2E-7 downgrade reason correction: epilepsy/ketogenic must match the supported conditions registry, and the user-facing explanation for a missing-slots downgrade must not tell users their condition is unsupported.
  • E2E-2 input payload restructure: explicitly provide the 6 non-medication slots as structured data so the slot heatmap reflects the scenario's actual intent.
  • 8 of 11 supported therapy conditions have zero eval scenarios. MSUD, galactosemia, food allergy, preterm nutrition, CKD, IBD, and GERD need dedicated scenarios before clinical coverage is complete.
  • Debug confidence and relevance pipeline connections so real values from the live CPNA API flow through correctly.

You cannot responsibly deploy clinical AI without eval infrastructure. The question is whether you build it before shipping or after an incident forces you to build it.

Every hour spent on this infrastructure protects the accuracy of every response CPNA will ever generate. For a clinical system serving pediatric patients, that protection is the product.
